# 대구 버스 실시간 지도 설계

- 작성일: 2026-07-17
- 상태: 사용자 승인 완료
- 대상: 출근·퇴근 대시보드에서 선택한 노선과 방향의 실시간 운행 차량 지도

## 1. 목적

대시보드의 출근·퇴근 카드를 누르면 선택한 노선과 방향의 실제 운행 차량을 지도에서 확인할 수 있게 한다. 지도는 정류소 사이를 단순 직선으로 잇지 않고 대구 버스 Open API의 노선 링크와 기초 노드를 결합한 링크 기반 경로를 사용한다.

이 기능의 최우선 원칙은 오래되거나 추정한 위치를 현재 위치처럼 보이지 않는 것이다. 차량은 `getPos02`가 확인해 준 GPS 좌표에만 표시하며, 새 정상 위치를 30초 동안 받지 못하면 지도에서 숨긴다.

## 2. 범위

### 포함

- 선택한 출근 또는 퇴근 카드의 노선·방향 전용 지도
- 네이버 지도 위의 링크 기반 노선선, 정류소 마커, 차량 마커
- 화면이 보이는 동안의 자동 차량 위치 갱신
- 지도 하단 시트의 노선·방향·내 정류장·차량 정보
- 정상, 빈 운행, 지연, 네트워크, 인증, 호출 한도, 지도 인증 오류 상태
- 노선 형상과 차량 스냅샷의 Room 캐시

### 제외

- 반대 방향 차량 표시
- 사용자 현재 위치와 위치 권한
- 차량 위치 보간·외삽과 도로 스냅 추정
- 백그라운드 차량 추적, 승차 알림, 도착 알람
- 실제 도로 곡선의 모든 중간 점을 포함하는 별도 GIS 형상 수집
- 차량 식별번호의 저장·표시·로그 기록

## 3. 사용자 흐름과 화면

1. 사용자가 대시보드의 출근 또는 퇴근 카드를 누른다.
2. 앱은 해당 카드에 저장된 `routeId`, `routeNo`, `moveDir`, 선택 정류소를 읽는다.
3. 캐시된 링크 기반 노선 형상이 있으면 즉시 지도에 표시한다. 캐시가 없으면 로딩 상태에서 노선 형상을 구성한다.
4. 첫 차량 위치 조회가 성공하면 선택 방향 차량만 표시하고, 노선과 차량을 포함하도록 최초 카메라 범위를 맞춘다.
5. 사용자가 카메라를 직접 움직인 뒤에는 자동 갱신이 카메라 위치를 되돌리지 않는다. 화면의 `노선 전체 보기` 동작으로만 다시 맞춘다.
6. 사용자가 뒤로 가거나 앱을 백그라운드로 보내면 차량 폴링을 즉시 중단한다.

화면은 승인된 `지도 중심 + 하단 시트` 구조를 사용한다.

### 지도

- 선택 방향의 링크 기반 노선선을 강조한다.
- 선택 방향의 정류소만 순서대로 표시한다.
- 즐겨찾기에 저장된 내 정류장은 다른 정류소보다 강하게 표시한다.
- 선택 방향의 현재 차량만 버스 마커로 표시한다.
- 차량 마커는 확인된 새 좌표로 즉시 교체한다. 마커 사이의 추정 애니메이션은 사용하지 않는다.

### 하단 시트

접힌 상태에서는 노선명, 방향, 내 정류장, 차량 수, 마지막 정상 수신 시각과 신선도 상태를 표시한다. 펼친 상태에서는 차량을 진행 순서로 정렬해 현재 정류소, 정류소 순번, 내 정류장까지 남은 정류장 수를 보여준다.

차량 마커를 누르면 해당 응답 스냅샷의 차량 항목을 하단 시트에서 강조한다. 차량 식별번호를 보존하지 않으므로 다음 갱신에서 동일 차량임을 추정하지 않으며 선택 강조를 해제한다.

## 4. 노선 형상

노선 형상은 다음 공식 데이터만 사용한다.

- `getLink02(routeId)`: `linkId`, `stNode`, `edNode`, `moveDir`, `linkSeq`
- `getBasic02`: 노드별 `nodeId`, `xPos`, `yPos`
- `getBs02(routeId)`: 방향별 정류소 좌표와 순서

`RouteGeometryRepository`는 선택한 `moveDir`의 링크를 `linkSeq`로 정렬하고 각 링크의 시작·종료 노드를 좌표로 변환한다. 인접 링크가 연결되도록 노드 방향을 정규화한 뒤 지도용 좌표열을 만든다. 링크 또는 노드 좌표가 누락되거나 인접성이 끊긴 구간은 하나의 직선으로 메우지 않고 별도 선분으로 분리한다. 잘못된 지름길을 그리지 않는 것이 완전한 연결선보다 우선한다.

이 형상은 정류소 2점을 바로 잇는 선보다 촘촘하게 도로망을 따르지만, API가 도로 곡선의 모든 중간 좌표를 제공하지 않으므로 실제 도로 곡률과 픽셀 단위로 일치한다고 보장하지 않는다.

완성된 방향별 좌표열과 구성 시각은 Room에 저장한다. 기존 기초 데이터 정책과 맞춰 24시간 동안 재사용하며, 갱신에 실패해도 유효한 캐시가 있으면 형상은 유지하고 형상 갱신 오류를 차량 위치 오류와 분리해 표시한다.

## 5. 차량 위치와 신선도

차량 위치는 `getPos02(routeId)`의 `moveDir`, `seq`, `bsId`, `xPos`, `yPos`, `arTime`을 사용한다. `xPos`는 경도, `yPos`는 위도다. 응답을 도메인 모델로 변환한 뒤 선택한 방향과 일치하는 항목만 UI에 전달한다.

신선도는 마지막 정상 GPS 스냅샷의 수신 시각을 기준으로 계산한다.

- `0~15초`: 정상. 차량 마커를 그대로 표시한다.
- `15초 초과~30초`: 지연. 마지막 확인 위치를 유지하되 하단 시트에 지연 상태와 데이터 나이를 표시한다.
- `30초 초과`: 오래됨. 차량 마커와 차량 목록을 숨기고 `위치 정보가 지연되고 있습니다`를 표시한다.

정상 빈 응답은 현재 운행 차량이 없는 상태로 취급한다. 이전 차량을 현재 운행 중인 것처럼 남기지 않고 즉시 마커와 목록을 비우며 `현재 운행 차량 없음`을 표시한다.

차량의 `vhcNo2`는 원격 파싱 경계에서 버린다. 도메인 모델, Room, UI 상태, 분석 정보와 로그에 포함하지 않는다. 따라서 서로 다른 응답 사이에서 차량 동일성을 추정하거나 부드러운 이동을 만들지 않는다.

## 6. 폴링과 수명주기

`RealtimeMapViewModel`은 지도 화면이 전경에서 보이는 동안 하나의 폴링 작업만 유지한다.

- 정상 응답 뒤: 8초 대기
- 첫 일시적 실패 뒤: 15초 대기
- 연속 두 번째 이후 일시적 실패 뒤: 30초 대기
- 인증 실패 또는 호출 한도 초과: 폴링 중단
- 화면 이탈 또는 앱 백그라운드 전환: 즉시 취소
- 화면 복귀: 기존 작업을 재사용하지 않고 새 단일 작업 시작

정상 8초 간격은 10분 동안 최초 호출을 포함해 최대 약 76회의 차량 위치 요청을 만든다. 기초·링크 데이터는 캐시해 반복 폴링에 포함하지 않는다. 폴링 작업과 사용자의 수동 재시도가 겹치면 저장소의 단일 비행 요청을 공유한다.

## 7. 구조와 책임

```text
RealtimeRouteMapScreen
    -> RealtimeMapViewModel
        -> RouteGeometryRepository
            -> DaeguBusRemoteDataSource (getBasic02, getLink02, getBs02)
            -> RoomLocalDataSource
        -> VehiclePositionRepository
            -> DaeguBusRemoteDataSource (getPos02)
            -> RoomLocalDataSource
        -> PollingPolicy / FreshnessPolicy
```

### `RouteGeometryRepository`

방향별 링크 조회, 노드 좌표 결합, 연결성 검증, 선분 분할과 Room 캐시를 담당한다. UI는 링크나 노드 원격 DTO를 직접 알지 않는다.

### `VehiclePositionRepository`

선택 노선 차량 조회, 방향 필터링, 정상·정상 빈 응답 구분, 마지막 정상 스냅샷과 수신 시각 저장을 담당한다. 오래된 위치를 표시할지 여부는 명시적 신선도 상태로 반환한다.

### `RealtimeMapViewModel`

화면 수명주기와 폴링을 연결하고 노선 형상 상태, 차량 상태, 하단 시트 표시 모델을 하나의 불변 `UiState`로 노출한다. 네트워크나 지도 SDK 객체를 상태에 직접 보관하지 않는다.

### `RealtimeRouteMapScreen`

네이버 지도와 Compose UI를 연결하고 노선선, 정류소·차량 마커, 하단 시트를 렌더링한다. 저장소를 직접 호출하지 않는다.

기존 일회성 차량 상세 목록은 별도 화면으로 유지하지 않고 지도 하단 시트의 펼침 목록으로 흡수한다.

## 8. 외부 의존성과 설정

네이버 지도 Android SDK의 구현 시점 공식 안정 버전을 사용한다. SDK 저장소는 이미 프로젝트 설정에 존재하며, 앱 패키지에 연결한 NCP Key ID는 ignored `local.properties`에서 빌드 설정으로 주입한다. 키 값은 소스, APK용 문서, 로그와 Git에 기록하지 않는다.

지도 인증 실패, 호출 한도, 네트워크 오류는 버스 API 오류와 구분한다. 지도 자체를 불러오지 못하면 빈 지도 위에 마커를 흉내 내지 않고 전체 오류 상태와 재시도·설정 안내를 제공한다.

## 9. 오류 처리

- 노선 형상 로딩 실패 + 캐시 있음: 캐시 형상을 표시하고 비차단 안내를 표시한다.
- 노선 형상 로딩 실패 + 캐시 없음: 차량 좌표만 지도에 흩어 표시하지 않고 재시도 가능한 전체 오류 상태를 표시한다.
- 차량 네트워크·서비스 오류: 마지막 정상 위치는 30초까지만 유지하고 이후 숨긴다.
- 정상 빈 차량 응답: 운행 없음 상태로 즉시 전환한다.
- 인증 오류: 폴링을 중단하고 API 키 변경 동작을 제공한다.
- 버스 API 호출 한도 초과: 폴링을 중단하고 한도 오류를 안내한다.
- 지도 인증·지도 호출 한도 오류: 지도 오류 상태로 전환하고 버스 API 폴링도 중단한다.
- 형식 오류 또는 유효 좌표가 없는 응답: 정상 스냅샷으로 저장하지 않는다.
- 네트워크 복구: 화면이 전경에 있으면 다음 재시도에서 정상 상태로 복귀한다.

오류 메시지와 로그에는 서비스 키, 전체 요청 URL, 원문 응답, 차량 식별번호를 포함하지 않는다.

## 10. 테스트

### 단위 테스트

- 방향별 링크를 `linkSeq`로 정렬하고 연결 노드 방향을 정규화한다.
- 누락 노드와 불연속 링크를 직선으로 건너지 않고 선분으로 분리한다.
- 반대 방향 차량을 제거하고 선택 방향 차량만 정렬한다.
- `vhcNo2`가 도메인·저장·UI 모델에 존재하지 않는다.
- 정상 빈 응답이 이전 차량 마커를 비운다.
- 가짜 시계로 15초 지연과 30초 숨김 경계를 검증한다.
- 정상 8초, 첫 실패 15초, 후속 실패 30초, 인증·한도 오류 중단을 검증한다.
- 화면 수명주기 이벤트가 중복 폴링을 만들지 않는다.

### UI 테스트

- 지도 로딩, 정상 차량, 빈 운행, 지연, 오래됨, 인증 오류와 지도 오류 상태를 렌더링한다.
- 하단 시트 접힘·펼침과 차량 마커 선택을 검증한다.
- 내 정류장과 일반 정류장의 시각적 구분을 검증한다.
- 노선 전체 보기 전까지 사용자 카메라 조작을 덮어쓰지 않는다.

### 실기기 테스트

- 급행8-1의 선택 방향만 표시되고 반대 방향 차량은 보이지 않는다.
- 링크 기반 노선선이 실제 도로망을 합리적으로 따라가며 불연속 구간을 가로지르지 않는다.
- 차량 마커가 공식 서비스와 비슷한 실제 위치에 나타난다.
- 네트워크 차단 후 데이터 나이가 증가하고 30초 뒤 차량 마커가 숨겨진다.
- 앱을 백그라운드로 보낸 뒤 차량 API 호출이 중단된다.
- 지도 인증 실패가 버스 API 키 오류로 잘못 표시되지 않는다.

연결 기기 테스트는 설치된 앱 데이터를 삭제할 수 있다. 사용자의 사전 동의 없이 `connectedDebugAndroidTest` 또는 동등한 전체 기기 테스트를 실행하지 않는다.

## 11. 완료 조건

1. 대시보드 카드에서 선택 노선·방향의 실시간 지도에 진입한다.
2. 링크·노드 기반 노선선과 선택 방향 정류소를 표시한다.
3. 선택 방향의 확인된 GPS 차량만 8초 간격으로 갱신한다.
4. 위치를 보간·외삽하거나 반대 방향 차량을 표시하지 않는다.
5. 새 정상 위치가 30초 동안 없으면 차량 마커와 목록을 숨긴다.
6. 화면 이탈과 백그라운드 전환 시 폴링이 중단된다.
7. 정상 빈 운행과 네트워크·인증·한도·지도 오류를 구분한다.
8. 서비스 키와 차량 식별번호가 화면, 상태, 로그와 Git에 포함되지 않는다.
9. 단위·UI 테스트가 통과하고, 사용자 동의 아래 실기기에서 급행8-1 동작을 확인할 수 있다.

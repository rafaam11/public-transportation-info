# 대구 버스 Android 앱 설계

- 작성일: 2026-07-15
- 상태: 사용자 승인 완료
- 대상: 개인 휴대폰에 직접 설치하는 1차 MVP

## 1. 목적

대구 버스정보시스템을 매일 사용하는 개인 사용자를 위해, 출근·퇴근 버스를 빠르게 확인하고 선택한 노선의 운행 차량 위치를 지도에서 추적할 수 있는 네이티브 Android 앱을 만든다.

기존 웹 서비스가 불안정해도 마지막 정상 위치를 지우지 않는다. 대신 데이터의 나이를 명확히 표시해 사용자가 최신 정보와 오래된 정보를 구분할 수 있게 한다. 도착 시간 예측보다 차량 위치 확인을 우선한다.

## 2. 제품 원칙

1. 마지막 위치를 숨기지 않는다.
2. 확인되지 않은 미래 위치를 추측하지 않는다.
3. 데이터 출처와 수신 시각을 사용자가 이해할 수 있게 표현한다.
4. 지도 화면이 보일 때만 자주 갱신한다.
5. 1차 MVP는 서버 없이 작동하되, 나중에 캐시 서버로 교체할 수 있는 경계를 둔다.
6. 위치 권한을 요청하지 않는다.

## 4. 기술 전제와 외부 의존성

### 4.1 대구 버스 Open API

공식 서비스는 JSON REST API로 기초 정보, 노선별 정류장, 노선별 링크, 정류장별 도착 예정 정보, 노선별 운행 차량 위치를 제공한다. 개발계정의 기본 호출 한도는 하루 1,000회다.

사용할 기능은 다음과 같다.

- `getBasic02`: 노선·정류장 기초 정보
- `getBs02`: 노선별 경유 정류장
- `getLink02`: 노선별 경유 링크 또는 경로 정보
- `getRealtime02`: 즐겨찾기 정류장의 도착·접근 차량 정보
- `getPos02`: 선택 노선의 운행 차량 위치

공식 API의 실제 응답 스키마와 좌표 정밀도는 발급받은 키로 구현 시작 전에 검증한다. 이 검증은 기능 구현의 선행 게이트다.

### 4.2 네이버 지도

네이버 지도 Android SDK를 사용한다. SDK용 NCP Key ID는 앱 패키지 이름에 연결하고 `local.properties`에서 빌드 설정으로 주입한다. 이 값은 버전 관리에 포함하지 않는다. 지도 인증 실패 코드 401, 429, 800은 앱의 지도 오류 상태로 변환한다.

지도 SDK는 현재 공식 가이드의 안정 버전을 우선한다. 설계 작성 시 공식 가이드는 `com.naver.maps:map-sdk:3.23.2`와 Android API 21 이상을 안내한다. 앱의 `minSdk`는 개인용 MVP의 유지보수 범위를 줄이기 위해 API 26으로 정한다.

### 4.3 Android UI와 위젯

- 앱 화면: Jetpack Compose와 단방향 데이터 흐름
- 위젯: 안정 버전의 Jetpack Glance
- 로컬 DB: Room
- 비동기 처리: Kotlin Coroutines와 Flow
- 일회성 위젯 갱신: 사용자 동작에서 시작하는 CoroutineWorker 또는 Glance action

Glance 위젯은 프로세스 메모리에 상태를 두지 않는다. 위젯의 표시 데이터는 Room과 영속 설정에서 읽는다.

## 5. 전체 아키텍처

```text
Compose screens ─┐
                 ├─ ViewModels ─ BusRepository ─┬─ DaeguBusRemoteDataSource
Glance widget ───┘                              ├─ RoomLocalDataSource
                                                ├─ CredentialStore
                                                └─ FreshnessPolicy / RequestCoordinator
```

UI와 위젯은 `BusRepository` 계약만 사용한다. 공식 API를 직접 아는 코드는 원격 데이터 소스에 한정한다. 2단계에서 개인용 캐시 서버를 추가할 때 `DaeguBusRemoteDataSource` 구현만 서버 API 구현으로 바꿀 수 있어야 한다.

### 5.1 주요 단위

#### Compose UI

대시보드, 최초 설정, 즐겨찾기 편집, 검색, 노선 지도, 설정을 렌더링한다. 상태를 직접 저장하거나 네트워크를 호출하지 않는다.

#### ViewModel

화면 이벤트를 받고 `BusRepository`를 호출하며 하나의 불변 `UiState`를 노출한다. 화면 생명주기에 따라 지도 폴링을 시작하고 중단한다.

#### BusRepository

캐시 우선 조회, 원격 최신화, 신선도 계산, 중복 요청 제거, 오류 통합을 담당한다. UI에는 원격 DTO가 아닌 도메인 모델과 명시적 데이터 상태를 반환한다.

#### DaeguBusRemoteDataSource

공식 API 요청, JSON 파싱, 응답 코드 분류, DTO 검증을 담당한다. 불완전하거나 형식이 바뀐 응답을 성공 데이터로 저장하지 않는다.

#### RoomLocalDataSource

노선, 정류장, 노선 순서·경로, 통근 카드, 차량 스냅샷, 마지막 성공 시각을 보존한다. 네트워크 장애와 프로세스 재시작 후에도 마지막 정상 상태를 제공한다.

#### CredentialStore

사용자가 설정 화면에 입력한 공공데이터 서비스 키를 앱 전용 저장소에 저장한다. 키를 로그, 크래시 메시지, 테스트 fixture, Git에 기록하지 않는다. 개인용 MVP이므로 루팅·역공학까지 막는 비밀 저장을 보장하지는 않는다. 공개 배포로 전환할 때는 키를 서버로 이동해야 한다.

#### RequestCoordinator

같은 노선 또는 정류장의 동시 요청을 하나로 합치고, 화면별 요청 횟수와 재시도 간격을 통제한다.

## 6. 도메인 모델

### CommuteFavorite

- `slot`: `COMMUTE_TO_WORK` 또는 `COMMUTE_TO_HOME`
- `stopId`, `stopName`
- `routeId`, `routeName`
- `directionId`, `directionLabel`
- `updatedAt`

슬롯마다 정확히 하나의 카드만 활성화할 수 있다. 카드는 앱 재시작 후에도 유지된다.

### VehicleSnapshot

- `routeId`
- `vehicleId`: 원천 데이터가 제공할 때 사용
- `directionId`
- `latitude`, `longitude`: 원천 데이터가 제공할 때만 사용
- `linkId`, `stopSequence`: 좌표가 없거나 보조 정보가 필요할 때 사용
- `sourceObservedAt`: 원천 응답이 제공할 때 사용
- `fetchedAt`: 앱이 성공 응답을 받은 시각

차량의 안정적인 식별자가 없으면 연속 애니메이션을 억지로 연결하지 않는다.

### DataFreshness

- `FRESH`: 데이터 나이 0~15초
- `DELAYED`: 15초 초과 30초 이하
- `STALE`: 30초 초과
- `UNAVAILABLE`: 저장된 정상 데이터가 한 번도 없음

경계는 `age <= 15s`, `15s < age <= 30s`, `age > 30s`로 해석한다. 원천 시각이 있으면 이를 우선하고, 없으면 `fetchedAt`을 사용한다.

## 7. 화면과 사용자 흐름

### 7.1 최초 설정

1. 앱이 저장된 서비스 키를 확인한다.
2. 키가 없으면 공공데이터포털 활용 신청 안내와 입력란을 표시한다.
3. 사용자가 키를 입력하면 작은 실제 요청으로 유효성을 검사한다.
4. 성공하면 출근 카드 설정으로 이동한다.
5. 정류장 검색 → 노선 선택 → 방향 확인 순서로 카드를 만든다.
6. 같은 방식으로 퇴근 카드를 만든다.
7. B형 대시보드로 이동한다.

키 검증에 실패하면 인증 실패, 네트워크 실패, 공식 서비스 장애를 구분해 표시한다. 네트워크 실패를 잘못된 키로 단정하지 않는다.

### 7.2 B형 대시보드

출근과 퇴근 카드를 같은 비중으로 세로 배치한다. 각 카드는 다음을 표시한다.

- 슬롯 이름과 정류장 이름
- 노선 이름과 방향
- 가장 가까운 접근 차량 정보
- 데이터 신선도와 경과 시간
- 지도 진입 동작

화면 진입 시 Room 데이터를 즉시 렌더링한 뒤 두 카드 각각을 한 번 최신화한다. 사용자는 화면 전체 새로고침을 다시 요청할 수 있다.

### 7.3 노선·정류장 검색

즐겨찾기와 무관하게 대구 노선이나 정류장을 검색할 수 있다. 노선을 선택하면 방향을 확인한 뒤 노선 지도로 이동한다. 검색 화면 자체는 실시간 데이터를 반복 호출하지 않는다.

### 7.4 선택 노선 지도

- 선택 노선의 경로와 모든 운행 차량을 표시한다.
- 지도 초기 카메라는 노선 전체 또는 현재 운행 차량 영역을 포함한다.
- 사용자 위치 버튼과 위치 권한 요청은 제공하지 않는다.
- 차량 마커에는 방향, 식별 정보, 마지막 수신 경과 시간을 표시한다.
- 새로 확인된 위치가 오면 이전 확인 위치에서 새 확인 위치까지만 짧게 애니메이션한다.
- stale 상태에서는 애니메이션을 중단하고 마커의 투명도를 낮춘다.
- 사용자가 지도를 떠나거나 앱이 백그라운드로 가면 폴링을 즉시 중단한다.

### 7.5 위젯

각 위젯 인스턴스는 출근 또는 퇴근 카드 중 하나를 선택한다. 위젯은 다음을 표시한다.

- 정류장과 노선
- 가장 가까운 접근 차량
- 마지막 정상 조회 시각과 신선도
- 새로고침 동작
- 앱의 해당 노선 지도를 여는 동작

위젯은 주기적 네트워크 갱신을 하지 않는다. 사용자가 새로고침을 누르면 선택된 카드만 한 번 요청하고, 성공하면 Room과 위젯을 갱신한다. 실패하면 마지막 데이터를 유지하고 오류 상태와 경과 시간을 갱신한다.

### 7.6 설정

- 공공데이터 서비스 키 입력, 재검증, 변경
- 출근·퇴근 카드 변경
- 마지막 정적 데이터·실시간 데이터 갱신 시각 확인
- 캐시 삭제
- 데이터 출처와 한계 확인

## 8. 데이터 흐름과 요청 정책

### 8.1 공통 캐시 우선 흐름

1. UI가 Repository의 Flow를 구독한다.
2. Repository가 Room의 마지막 정상 데이터를 즉시 반환한다.
3. 화면 정책이 허용할 때 원격 최신화를 시작한다.
4. 원격 응답의 상태 코드와 필수 필드를 검증한다.
5. 정상 응답만 하나의 트랜잭션으로 Room에 저장한다.
6. Room Flow가 변경된 상태를 UI와 위젯에 전달한다.

### 8.2 화면별 호출

- 대시보드: 진입 시 즐겨찾기 정류장 2개를 각각 한 번 조회
- 노선 지도: 진입 즉시 한 번 조회한 뒤 정상 상태에서 8초 간격
- 위젯: 사용자의 새로고침 동작당 선택 카드 한 번 조회
- 기초 정보: 사용자가 앱을 열었고 저장 데이터가 24시간보다 오래됐을 때 한 번 갱신
- 검색: 저장된 기초 정보를 사용하며 검색 입력마다 네트워크를 호출하지 않음

같은 키의 진행 중 요청이 있으면 새 요청을 만들지 않고 기존 결과를 공유한다.

### 8.3 지도 재시도

- 성공: 다음 요청까지 8초
- 연속 첫 실패: 다음 요청까지 15초
- 두 번째 이후 실패: 다음 요청까지 30초
- 인증 실패 또는 한도 초과: 자동 폴링 중단
- 화면 이탈 또는 백그라운드: 즉시 취소

성공하면 재시도 간격을 8초로 복구한다. 자동 재시도는 동시에 하나만 존재한다.

### 8.4 호출 예산

10분 지도 세션은 최초 요청을 포함해 최대 약 76회의 위치 요청을 사용한다. 하루 두 번의 10분 세션, 대시보드 진입, 수동 위젯 갱신, 기초 정보 갱신을 합쳐도 개인 사용 기준으로 개발계정 1,000회 한도 안에 충분한 여유를 둔다. 앱은 일별 요청 횟수를 로컬에 기록해 설정 화면에서 확인할 수 있게 한다.

## 9. 장애와 불완전 데이터 처리

### 9.1 네트워크 시간 초과와 서버 오류

마지막 정상 데이터를 유지한다. 화면에는 오류 배너와 데이터 나이를 표시하고 지도 재시도 간격을 늘린다.

### 9.2 인증 실패

자동 폴링을 중단하고 설정 화면의 키 검증으로 안내한다. 저장된 데이터는 삭제하지 않는다.

### 9.3 호출 한도 초과

자동 요청을 중단한다. 사용량 제한 상태와 마지막 정상 데이터를 표시한다. 반복 새로고침으로 제한을 악화시키지 않는다.

### 9.4 정상 빈 응답

운행 시간이 아니거나 해당 노선에 차량이 없는 정상 응답은 장애와 구분해 “현재 운행 차량 없음”으로 표시한다. 이전 차량을 현재 운행 중인 것처럼 유지하지 않으며, 과거 스냅샷은 상태 설명용으로만 남긴다.

### 9.5 부분·비정상 응답

필수 식별자나 위치 정보가 없는 항목은 격리한다. 응답 전체가 신뢰할 수 없으면 기존 정상 스냅샷을 유지한다. 원문 응답이나 서비스 키를 사용자 로그에 저장하지 않는다.

### 9.6 첫 정상 데이터가 없는 실패

빈 지도나 무한 로딩을 보여주지 않는다. 실패 원인, 재시도, API 키 설정 이동을 제공한다.

### 9.7 위치 정밀도 제한

`getPos02`가 위도·경도를 제공하면 해당 좌표를 사용한다. 링크 또는 정류장 순서만 제공하면 공식 노선 경로상의 해당 구간으로 표시하고 “구간 기반 위치”임을 설명한다. 두 경우 모두 다음 위치를 예측해 계속 이동시키지 않는다.

## 10. 테스트 전략

### 10.1 API 계약 테스트

- 정상 응답 fixture
- 운행 차량이 없는 빈 응답
- 필수 필드 누락과 타입 변경
- 인증 실패와 호출 한도 오류
- 시간 초과와 서버 오류
- 로컬 서비스 키가 있을 때만 실행하는 opt-in 실시간 계약 확인

fixture에는 실제 서비스 키와 개인 식별 가능 값이 없어야 한다.

### 10.2 Repository 단위 테스트

- 캐시를 원격보다 먼저 방출
- 성공 응답만 저장
- 실패 시 마지막 정상 데이터 유지
- 같은 키의 동시 요청을 하나로 통합
- 8초, 15초, 30초 재시도 전이
- `age <= 15s`, `15s < age <= 30s`, `age > 30s` 경계
- 인증과 한도 오류에서 폴링 중단
- 화면 취소 시 진행 중 요청 취소

테스트는 가짜 시계와 가짜 데이터 소스를 사용해 실제 시간이나 네트워크에 의존하지 않는다.

### 10.3 Room 테스트

- 통근 카드 2개 저장·교체
- 차량 스냅샷 트랜잭션 교체
- 프로세스 재시작 후 데이터 복구
- 스키마 마이그레이션

### 10.4 UI와 위젯 테스트

- 최초 설정과 API 키 상태별 화면
- B형 대시보드의 두 카드
- 카드에서 올바른 노선·방향 지도로 이동
- 정상·지연·stale 마커 표현
- 위치 권한 요청이 존재하지 않음
- 위젯 인스턴스별 카드 선택
- 위젯 수동 새로고침 성공과 실패
- 위젯 탭으로 해당 노선 지도 열기

### 10.5 실기기 검증

- 비행기 모드에서 마지막 위치 유지
- 느린 네트워크에서 중복 요청 없음
- 백그라운드 전환 즉시 폴링 중단
- 프로세스 종료 후 캐시 복구
- 잘못된 버스 API 키와 네이버 지도 Client ID 오류
- 10분 지도 세션 호출 횟수
- 홈 화면 위젯 크기와 수동 갱신

## 11. 완료 기준

1. 앱에서 공공데이터 서비스 키를 입력하고 유효성을 검증할 수 있다.
2. 출근·퇴근 카드 2개가 앱 재시작 뒤에도 유지된다.
3. 대시보드는 Room 데이터를 즉시 표시한 뒤 각 카드를 한 번 최신화한다.
4. 선택 노선의 모든 현재 운행 차량을 지도에 표시한다.
5. 차량 마커는 새로 확인된 위치까지만 부드럽게 이동한다.
6. 30초를 넘은 데이터는 stale 상태와 경과 시간을 표시한다.
7. 네트워크 실패 시 마지막 정상 위치가 사라지지 않는다.
8. 정상 빈 응답은 “운행 차량 없음”으로 표시한다.
9. 위젯은 사용자가 눌렀을 때만 선택 카드 하나를 갱신한다.
10. 앱은 위치 권한을 요청하지 않는다.
11. 10분 지도 세션의 위치 요청은 76회를 넘지 않는다.
12. 지도 화면 이탈과 백그라운드 전환 시 폴링이 중단된다.
13. 서비스 키, 지도 키, 서명 키가 Git에 포함되지 않는다.

## 12. 구현 순서와 게이트

1. Git·Android 프로젝트 기반과 로컬 비밀 설정 경계
2. 공식 API 활용 신청과 실제 응답 캡처
3. `getPos02` 위치 정밀도·갱신 주기·차량 식별자 검증
4. 계약 fixture와 도메인 모델
5. Room과 Repository의 캐시 우선 동작
6. 최초 설정과 두 카드 대시보드
7. 네이버 지도와 선택 노선 폴링
8. stale 표시와 부드러운 확인 위치 애니메이션
9. Glance 위젯과 수동 갱신
10. 장애 시나리오와 실기기 검증

3번 게이트에서 GPS 좌표가 없다는 사실이 확인되더라도 구현을 중단하지 않는다. 대신 지도 표현을 구간 기반 위치로 한정하고 UI에 정밀도 한계를 표시한다.

## 13. 2단계 확장

개인용 MVP에서 실제 효용이 확인되면 캐시 서버를 추가한다. 서버는 공식 API 키 보호, 요청 통합, 장애 시 마지막 정상 데이터 제공, 다중 사용자 호출량 관리를 담당한다. 앱의 UI·ViewModel·도메인 모델은 유지하고 Repository 뒤의 원격 데이터 소스만 서버 API 구현으로 교체한다.

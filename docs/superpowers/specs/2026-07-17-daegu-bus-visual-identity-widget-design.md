# 대구버스 시각 아이덴티티·위젯 설계

- 날짜: 2026-07-17
- 상태: 사용자 승인 완료
- 대상: 실시간 지도 차량 마커, Android 앱 아이콘, 출근·퇴근 홈 화면 위젯

## 1. 목적

현재 실시간 지도에서 숫자나 정면 버스 아이콘처럼 보이는 차량 마커를, 노선명과 계통색을 직접 읽을 수 있는 옆면 버스 픽토그램으로 교체한다. 같은 조형 언어를 앱 아이콘과 홈 화면 위젯에도 적용해 앱 전체에 일관된 정체성을 만든다.

사용자가 확인되지 않은 차량 위치나 데이터 신선도를 오해하지 않게 하는 기존 원칙은 유지한다. 시각 개선은 정보의 정확성을 희생하지 않는다.

## 2. 범위

이번 단계에 포함한다.

- 노선 진행 방향에 맞춰 회전하는 옆면 차량 마커
- 차량 옆면에 표시되는 전체 노선명
- 대구버스 노선 타입에 따른 계통색
- 선택 차량 강조와 수평 도착정보 캡션
- 같은 픽토그램 언어를 사용하는 Adaptive Icon과 단색 테마 아이콘
- 출근 또는 퇴근 카드 하나를 선택하는 Android 홈 화면 위젯
- 위젯 수동 새로고침과 해당 실시간 지도 열기

이번 단계에 포함하지 않는다.

- 교통 정체 노선선
- 실시간 신호등 상태와 잔여시간
- 위젯의 주기적 백그라운드 네트워크 갱신
- 차량 위치 보간, 예측 또는 도로 위 강제 스냅

교통 정체는 다음 별도 단계, 실시간 신호는 대구 커버리지 탐색을 포함하는 그 다음 별도 단계로 설계한다.

## 3. 시각 방향

사용자가 선택한 방향은 `B. 교통 픽토그램형`이다.

- 작은 지도에서도 버스 옆면으로 즉시 인식되는 단순한 둥근 차체
- 흰색 창문, 바퀴 두 개, 계통색 차체
- 차체 중앙의 굵은 노선명
- 사실적인 문, 램프, 도색선은 생략
- 노선색은 장식이 아니라 노선 타입을 전달하는 정보로 사용
- 앱의 기존 밝은 바탕과 대구 파란색 상호작용 색은 유지

이 앱을 기억하게 하는 시각적 서명은 `계통색 차체에 노선명이 적힌 옆면 버스`다.

## 4. 노선 타입과 색상

### 4.1 데이터 원천

대구버스 `getBasic02` 응답의 `routeTCd`를 nullable `routeTypeCode`로 `RouteSummary`, `FavoriteSelection`, Room의 `RouteEntity`와 `FavoriteEntity`에 저장한다. 노선명 접두어를 파싱해 타입을 추측하지 않는다. 즐겨찾기를 생성할 때 선택한 노선의 타입을 함께 복사하므로 앱 재시작 직후에도 지도와 위젯이 같은 색을 사용한다.

현재 대구버스 웹 화면이 사용하는 타입색을 앱의 기준값으로 사용한다.

| `routeTCd` | 유형 | 차체색 | 글자색 |
|---|---|---|---|
| `1` | 급행 | `#FF4917` | 흰색 |
| `2` | 순환 | `#5BD338` | 검정 |
| `3` | 간선 | `#2C78CF` | 흰색 |
| `4` | 지선 | `#FFC000` | 검정 |
| `5`, `6`, `7` | 기타 지역 유형 | `#4330D6` | 흰색 |
| 그 외 | 미확인·외부 유형 | `#306FD9` | 흰색 |

색상표는 `RouteColorResolver` 한 곳에서만 관리한다. 현재 API에서 관찰되는 `91`, `C`, `G`, `Y`는 의미와 공식색을 검증하기 전까지 대구 파란색을 사용한다. 타입이 새로 추가되어도 임의의 색을 추측하지 않는다.

### 4.2 대비

색상별 전경색은 표에 명시하되 렌더러도 최소 대비를 검사한다. 밝은 순환·지선 차체에는 검정 글자, 급행·간선·기타·기본 차체에는 흰색 글자를 쓴다. 외곽선과 그림자는 지도 배경과의 분리를 위한 보조 수단이며 타입색 자체를 바꾸지 않는다.

## 5. 차량 마커

### 5.1 모양과 크기

- 기준 크기: 약 `72 x 34dp`
- 구성: 둥근 차체, 창문 영역, 바퀴 두 개, 노선명
- 노선명: 접미사를 포함한 전체 문자열
- 기본 마커: 얇은 외곽선과 짧은 그림자
- 선택 마커: 약 15% 확대, 흰색 강조 외곽선, 높은 `zIndex`
- 도착정보 캡션: `3정거장 전`, `도착`, `통과`를 기존처럼 표시

노선명이 길면 미리 정한 단계로 글자 크기를 줄인다. 구분에 중요한 `-1`, `-7` 같은 접미사를 생략하거나 말줄임표로 바꾸지 않는다. 최소 크기에서도 들어가지 않는 비정상 문자열은 마커에 가능한 전체 문자열을 표시하고 수평 캡션에 전체 노선명을 추가한다.

### 5.2 생성 방식

`BusMarkerRenderer`가 Android Canvas로 Bitmap을 생성한다. 입력은 노선명, 계통색, 전경색, 선택 상태, 화면 밀도다.

`BusMarkerIconCache`는 다음 키로 결과를 재사용한다.

```text
routeNo + routeColor + selected + density
```

같은 노선의 여러 차량은 Bitmap 하나를 공유한다. 5초 폴링 때 이미지를 다시 그리지 않고 좌표, 각도, 캡션, 선택 상태만 갱신한다.

### 5.3 진행 각도

`VehicleHeadingResolver`는 차량의 확인 좌표와 가장 가까운 선택 방향 노선 구간을 찾고 그 구간의 진행 각도를 계산한다.

- 위치는 API 좌표 그대로 둔다.
- 노선 위로 좌표를 이동시키지 않는다.
- 다음 위치를 예측하지 않는다.
- 차량과 후보 구간의 거리가 신뢰 한계를 넘으면 각도를 적용하지 않는다.
- 각도를 적용하면 차체와 차체 내부 노선명이 함께 회전한다.
- 도착정보 캡션은 항상 수평을 유지한다.

노선 데이터가 없거나 구간 진행 순서를 판별할 수 없는 경우 기본 각도 0도를 사용한다. 부정확한 방향을 확정적으로 표현하지 않는다.

### 5.4 겹침과 신선도

여러 차량이 가까워도 실제 위치를 보존하기 위해 마커를 임의로 벌리지 않는다. 선택 마커만 크기와 표시 순서로 구분한다.

기존 신선도 정책을 유지한다.

- 마지막 정상 응답이 stale이면 차량 마커를 표시하지 않는다.
- 실패 응답으로 마지막 정상 좌표를 새 위치처럼 다시 표시하지 않는다.
- 화면이 백그라운드로 가면 폴링을 중단한다.

## 6. 앱 아이콘

Adaptive Icon은 짙은 대구 파란색 배경과 흰색 옆면 버스 픽토그램을 사용한다. 버스 아래에는 급행·순환·지선을 상징하는 짧은 빨강·초록·노랑 띠를 넣어 여러 계통을 다루는 앱임을 표현한다.

- 아이콘에는 특정 노선번호를 넣지 않는다.
- 원형, 둥근 사각형, 스퀘어 마스크의 안전영역에서 차체와 바퀴가 잘리지 않아야 한다.
- Android 13 이상 테마 아이콘용 단색 foreground를 제공한다.
- 작은 런처 크기에서 사라지는 문, 램프, 세부 도색은 넣지 않는다.

## 7. 홈 화면 위젯

### 7.1 인스턴스와 크기

각 위젯 인스턴스는 `출근` 또는 `퇴근` 중 하나를 선택한다. 사용자는 두 위젯을 각각 배치할 수 있다. 기본은 작은 카드형 위젯이며 시스템 크기 변화에 따라 간격과 보조 문구를 줄인다.

### 7.2 정보 구조

```text
출근                              3초 전
[계통색 배지] 급행8-1
도착 임박
진천역(1번출구) · 유곡리공영차고지건너 방면
                              새로고침
```

- 상단: 출근·퇴근 슬롯과 마지막 정상 데이터 경과시간
- 중심: 계통색 노선 배지와 가장 가까운 차량의 도착 상태
- 하단: 정류장과 방향
- 동작: 카드 본문은 해당 실시간 지도 열기, 새로고침은 한 번의 원격 조회

위젯에는 지도를 넣지 않는다. 움직이지 않는 지도 마커가 실시간처럼 보이는 문제와 홈 화면 복잡도를 피한다.

### 7.3 갱신 정책

- 주기적 백그라운드 네트워크 갱신 없음
- 앱에서 성공한 최신 조회 결과는 Room을 통해 위젯과 공유
- 사용자가 새로고침을 누르면 선택된 카드만 한 번 요청
- 요청 중에는 중복 요청을 막고 진행 상태 표시
- 실패하면 마지막 정상 값을 유지하고 `갱신 실패 · 12분 전`처럼 표시
- 정상 데이터가 없으면 `아직 받은 정보 없음`과 새로고침 동작 제공

Android Glance를 사용하고 앱의 Repository와 Room을 데이터 원천으로 삼는다. 위젯이 별도의 네트워크·파싱 규칙을 갖지 않는다.

## 8. 컴포넌트 경계와 데이터 흐름

```text
getBasic02.routeTCd
  -> RouteSummary / RouteEntity
  -> FavoriteSelection / FavoriteEntity
  -> 지도 UI 상태와 위젯 상태
  -> RouteColorResolver
  -> BusMarkerRenderer / Widget route badge

vehicle confirmed coordinate + direction-specific RouteGeometry
  -> VehicleHeadingResolver
  -> NAVER Marker position + angle

Repository / Room snapshot
  -> dashboard
  -> widget state
  -> manual widget refresh
```

각 컴포넌트의 책임은 다음과 같다.

- `RouteColorResolver`: 타입 코드에서 의미색과 전경색 결정
- `BusMarkerRenderer`: 순수 렌더링 입력에서 Bitmap 생성
- `BusMarkerIconCache`: 렌더 결과 재사용과 수명 관리
- `VehicleHeadingResolver`: 확인 좌표 주변의 신뢰 가능한 노선 각도 계산
- `NaverMapOverlayController`: 마커 생성·갱신·선택·정리
- 위젯 상태 매퍼: 저장된 출근·퇴근 스냅샷을 Glance 표시 모델로 변환
- 위젯 갱신 작업: 명시적 사용자 요청 한 건 실행 후 Room과 위젯 갱신

## 9. 오류 처리

- 알 수 없는 `routeTCd`: 기본 파란색, 기능 계속
- Room의 기존 노선·즐겨찾기 행에 타입 없음: nullable 마이그레이션 후 기본색 사용. 다음 정적 노선 갱신에서 `routeId`로 즐겨찾기 타입까지 보완
- 노선 형상 없음: 마커 표시 가능, 회전 없음
- 신뢰 가능한 인접 구간 없음: 마커 표시 가능, 회전 없음
- Bitmap 생성 실패: 기존 단색 버스 drawable로 대체
- 위젯 네트워크 실패: 마지막 정상 데이터 유지
- 위젯 인증 오류: API 키 변경 화면을 여는 동작 제공
- 위젯 대상 카드 삭제·변경: 재설정 안내 상태 표시

## 10. 테스트

### 10.1 단위·저장소 테스트

- `routeTCd` 파싱
- 기존 Room 데이터가 보존되는 스키마 마이그레이션
- 타입 코드별 색상과 전경색
- 알 수 없는 코드 기본색
- 직선, 굴곡, 역방향 구간의 각도
- 노선과 너무 먼 차량의 각도 거부
- Bitmap 크기, 투명 여백, 긴 노선명 경계
- 캐시 키와 재사용
- stale 상태에서 차량 마커 미생성
- 위젯 상태 매핑, 수동 갱신, 마지막 정상값 유지

### 10.2 UI·통합 테스트

- 선택 차량 확대와 `zIndex`
- 위젯 본문으로 해당 지도 열기
- 새로고침 중 중복 요청 방지
- 빈 상태, 실패 상태, stale 상태 문구
- Adaptive Icon 리소스와 테마 아이콘 존재 검증

### 10.3 실기기 승인 기준

급행8-1 출근 방향에서 다음을 확인한다.

1. 모든 정상 차량이 빨간 옆면 버스로 보인다.
2. 각 마커에서 `급행8-1` 전체 문자열을 읽을 수 있다.
3. 버스와 노선명이 인접 도로의 진행 각도로 함께 회전한다.
4. 차량 좌표가 회전을 위해 이동하지 않는다.
5. 차량 선택과 하단 상세정보가 기존처럼 동작한다.
6. stale 상태에서는 마커가 숨겨진다.
7. 런처에서 새 Adaptive Icon과 테마 아이콘이 올바르게 보인다.
8. 출근 위젯에서 최신 도착정보를 보고 수동 새로고침할 수 있다.
9. 위젯을 누르면 급행8-1 출근 방향 실시간 지도가 열린다.
10. 단위 테스트, 계측 테스트, debug 빌드가 통과한다.

## 11. 후속 단계

이 설계 구현과 실기기 검증 후 다음 두 기능을 별도 설계한다.

1. 대구 실시간 구간속도를 노선 링크에 매칭한 원활·서행·정체 표시
2. 전국 통합 신호 API의 대구 커버리지 탐색 후 진행 방향별 신호 상태·잔여시간 표시

두 후속 기능도 오래된 값을 현재 상태처럼 보이지 않는 동일한 신선도 원칙을 적용한다.

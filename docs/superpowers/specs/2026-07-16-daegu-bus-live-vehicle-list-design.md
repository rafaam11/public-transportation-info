# 대구 버스 실시간 차량 목록 설계

- 작성일: 2026-07-16
- 상태: 사용자 승인 완료
- 대상: API 키 입력부터 814번 실시간 차량 목록까지 연결하는 첫 실사용 APK

## 1. 목적

현재의 정적 준비 화면을 실제 데이터가 보이는 첫 번째 세로형 기능으로 교체한다. 사용자는 공공데이터포털 서비스 키를 앱에서 입력하고 검증한 뒤, 814번 노선의 현재 운행 차량을 목록으로 확인하고 직접 새로고침할 수 있다.

이번 단계는 API 연결과 오류 처리의 실기기 검증에 집중한다. 출근·퇴근 카드, 노선·정류장 검색, 네이버 지도, Room, 위젯은 다음 구현 단계로 분리한다.

## 2. 사용자 흐름

1. 저장된 서비스 키가 없으면 키 입력 화면을 표시한다.
2. 사용자가 키를 제출하면 `getBasic02`를 호출해 HTTP 상태와 JSON 응답 헤더를 함께 검증한다.
3. `header.resultCode=0000`이고 `header.success=true`이면 키를 앱 전용 저장소에 저장한다.
4. 검증 성공 직후 814번(`routeId=3000814001`)의 `getPos02`를 호출한다.
5. 화면에 노선 번호, 현재 차량 수, 마지막 정상 수신 시각과 차량 목록을 표시한다.
6. 사용자가 새로고침을 누르면 `getPos02`를 한 번 호출한다. 자동 폴링은 하지 않는다.
7. 사용자는 저장된 키를 지우고 다시 입력할 수 있다.

## 3. 화면 설계

### 3.1 API 키 입력

- 앱 이름과 이 단계의 목적을 설명한다.
- 서비스 키 입력란은 기본적으로 내용을 가린다.
- 검증 중에는 중복 제출을 막고 진행 상태를 표시한다.
- 인증 오류, 네트워크 오류, 공식 서비스 오류를 서로 다른 문구로 안내한다.
- 네트워크 실패를 잘못된 키로 단정하지 않는다.

### 3.2 814번 차량 목록

- 상단에 `814번 실시간 차량` 제목과 키 변경 동작을 둔다.
- 요약 영역에 운행 차량 수, 마지막 정상 수신 시각, 데이터 신선도를 표시한다.
- 각 차량 항목에는 방향(`moveDir`), 현재 정류장 ID 또는 정류장 순서(`bsId`, `seq`), 좌표(`yPos`, `xPos`), 도착 관련 원천 값(`arTime`)을 표시한다.
- `vhcNo2`는 파싱 경계에서 버리고 저장, 상태, 화면, 로그에 포함하지 않는다.
- 목록은 방향과 정류장 순서를 기준으로 안정적으로 정렬한다.
- 새로고침 버튼은 한 번의 요청만 만든다.

### 3.3 상태 표현

- 첫 조회 전: 짧은 로딩 상태를 표시한다.
- 정상 데이터: 목록과 마지막 수신 시각을 표시한다.
- 정상 빈 응답: `현재 운행 차량 없음`을 표시하며 장애로 취급하지 않는다.
- 새로고침 실패와 이전 정상 데이터 존재: 이전 목록을 유지하고 상단 오류 배너와 데이터 나이를 표시한다.
- 첫 정상 데이터가 없는 실패: 오류 원인, 재시도, 키 변경 동작을 표시한다.
- 신선도는 기존 정책대로 `0~15초 FRESH`, `15초 초과~30초 DELAYED`, `30초 초과 STALE`을 사용한다.

## 4. 구조와 책임

```text
Compose screen
    -> VehicleListViewModel
        -> BusRepository
            -> DaeguBusRemoteDataSource
            -> CredentialStore
```

### CredentialStore

Android 앱 전용 설정 저장소에 서비스 키를 보관하고 읽고 삭제한다. 키를 로그나 오류 메시지에 넣지 않는다. 개인 설치용 APK의 로컬 보관이며 공개 배포 수준의 비밀 보호를 보장하지 않는다.

### DaeguBusRemoteDataSource

`getBasic02`와 `getPos02` 요청, JSON 역직렬화, 공통 응답 헤더 검증, 오류 분류를 담당한다. HTTP 200이어도 `resultCode` 또는 `success`가 실패이면 성공 데이터로 반환하지 않는다.

### BusRepository

키 검증과 814번 차량 조회를 제공한다. 프로세스가 살아 있는 동안 마지막 정상 차량 목록을 보존하고, 후속 요청 실패 시 이를 삭제하지 않는다. 정상 빈 응답은 성공 상태로 반환하되 이전 차량을 현재 운행 중인 것처럼 노출하지 않는다.

### VehicleListViewModel

화면 이벤트를 직렬화하고 하나의 불변 `UiState`를 노출한다. 초기 키 확인, 키 검증, 차량 조회, 수동 새로고침, 키 삭제를 조정한다. 동시에 두 개의 검증 또는 차량 요청을 실행하지 않는다.

### Compose UI

네트워크나 저장소를 직접 호출하지 않고 `UiState`를 렌더링하며 사용자 이벤트만 ViewModel에 전달한다. 검은 기본 배경 대신 Material 3 색상과 여백을 적용한 전체 화면을 제공한다.

## 5. 데이터 계약

Base URL은 `https://apis.data.go.kr/6270000/dbmsapi02`다.

- 키 검증: `GET /getBasic02?serviceKey=<key>`
- 차량 조회: `GET /getPos02?serviceKey=<key>&routeId=3000814001`
- 성공 조건: HTTP 성공과 함께 `header.resultCode=0000`, `header.success=true`
- 차량 입력 필드: `routeId`, `routeNo`, `moveDir`, `arTime`, `seq`, `bsId`, `xPos`, `yPos`, `busTCd2`, `busTCd3`
- 민감 필드: `vhcNo2`는 DTO의 보존 가능한 속성으로 정의하지 않는다.
- 좌표 변환: `xPos`는 경도, `yPos`는 위도다.

원격 DTO는 도메인 모델로 변환한 뒤 UI로 전달한다. 필수 좌표나 노선 식별자가 없거나 숫자로 변환할 수 없는 차량 항목은 제외한다. 응답 전체가 계약을 충족하지 못하면 이전 정상 상태를 유지한다.

## 6. 오류 분류

- `InvalidCredential`: 인증 또는 필수 파라미터 오류. 자동 재시도하지 않고 키 변경을 안내한다.
- `RateLimited`: 호출 한도 초과. 반복 요청을 막고 제한 상태를 표시한다.
- `NetworkUnavailable`: 연결 실패 또는 시간 초과. 재시도 동작을 제공한다.
- `ServiceUnavailable`: 서버 오류 또는 정상 형식이 아닌 응답. 이전 데이터를 유지한다.
- `MalformedResponse`: 성공으로 표시됐지만 필수 구조가 없는 응답. 저장하지 않는다.

화면과 로그에는 서비스 키, 전체 요청 URL, 원문 응답, `vhcNo2`를 기록하지 않는다.

## 7. 저장과 캐시 경계

- 서비스 키는 앱 재시작 후에도 유지한다.
- 차량 목록은 이번 단계에서 메모리에만 캐시한다.
- 프로세스 재시작 뒤 차량 목록을 복원하는 Room 캐시는 출근·퇴근 카드 및 지도 스냅샷 단계에서 함께 도입한다.
- 이 경계는 향후 `BusRepository`의 소비자 계약을 바꾸지 않고 영속 캐시를 추가할 수 있게 한다.

## 8. 테스트

### 단위 테스트

- 성공 envelope와 814번 차량 배열을 도메인 모델로 변환한다.
- HTTP 200 내부의 실패 헤더를 오류로 분류한다.
- `vhcNo2`가 도메인 모델과 UI 상태에 존재하지 않는다.
- 좌표 누락 차량을 제외한다.
- 정상 빈 응답을 성공 빈 목록으로 처리한다.
- 후속 실패 시 마지막 정상 목록을 유지한다.
- 동시 새로고침을 한 요청으로 제한한다.
- 기존 freshness 경계 테스트를 유지한다.

### UI 테스트

- 저장 키 유무에 따라 입력 화면과 차량 화면을 전환한다.
- 검증 중 중복 제출을 차단한다.
- 정상, 빈 운행, stale, 인증 오류, 네트워크 오류를 구분해 표시한다.
- 키 삭제 후 입력 화면으로 돌아간다.

### 실기기 테스트

- 실제 키 입력 후 814번 운행 차량을 표시한다.
- 새로고침으로 마지막 정상 수신 시각이 갱신된다.
- 차량 목록을 표시한 뒤 네트워크를 끊어도 목록이 사라지지 않는다.
- 잘못된 키가 인증 오류로 표시된다.
- 앱 재실행 후 저장된 키로 차량 화면에 진입한다.

## 9. 완료 조건

1. 설치된 앱에서 서비스 키를 입력하고 검증할 수 있다.
2. 유효한 키는 앱 재실행 후에도 유지된다.
3. 814번의 실제 운행 차량 수와 차량 목록을 표시한다.
4. 수동 새로고침은 동시 요청을 만들지 않는다.
5. 새로고침 실패 시 마지막 정상 목록과 데이터 나이를 유지한다.
6. 정상 빈 응답과 장애를 구분한다.
7. `vhcNo2`와 서비스 키가 화면, 상태, 로그, Git에 포함되지 않는다.
8. 기존 단위 테스트와 새 단위·UI 테스트가 통과한다.
9. 새 debug APK를 실기기에 설치해 정상·오류 흐름을 확인할 수 있다.

## 10. 후속 단계

이 단계가 실기기에서 통과하면 Room 영속 캐시와 출근·퇴근 카드 대시보드를 구현한다. 그 다음 네이버 지도, 위치 폴링과 stale 마커, Glance 위젯 순서로 확장한다.
